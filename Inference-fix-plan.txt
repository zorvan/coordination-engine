Diagnosis: Why "I inferred an event then stops"
The bot's UX failure has four compounding root causes:
Root Cause 1 — max_tokens: 500 is critically too low
In ai/llm.py, every LLM call is capped at 500 tokens. The infer_event_draft_from_context prompt is large (it includes 30 messages of chat history + the prompt template itself), leaving the LLM almost no room to output a full event JSON. The response gets truncated mid-JSON, json.loads() throws, and the fallback silently fires — which produces a skeleton draft with description = message_text but no rich inferred details.
Fix: Raise max_tokens to at least 1500 for draft inference calls, and use a separate _call_llm_large() helper for context-heavy calls.

Root Cause 2 — _handle_organize_event_direct shows nothing meaningful on the happy path
When the LLM does produce a valid description and the event gets created, the group message is a raw data dump (Event ID, type, commit-by, budget, transport, date preset, time window…). It reads like a database row, not a social invitation. Users see this wall of technical text and have no idea what to do next. The bot also sends the full admin summary only to the organizer DM, but group members get no call-to-action buttons.
Fix: Replace the group creation message with a warm, narrative-style announcement that includes one-tap Join/Confirm buttons, clearly shows what's missing (time if flexible), and uses 🌱 → 🎉 visual language from the PRD's materialization layer.

Root Cause 3 — The opinion fallback is a dead-end
When the LLM returns action_type: "opinion" (its default when uncertain), the bot emits whatever assistant_response is in the JSON — often a vague AI-generated advisory — and stops. There's no follow-up offer, no inline keyboard, no path forward for the user. In tight situations (ambiguous context, no event ID, sparse history), this happens constantly.
Fix: The opinion branch needs to detect intent ambiguity vs genuine advisory request, and for the former, render an interactive disambiguation keyboard: [🗓 Create Event] [📋 View Events] [❓ Ask a question].

Root Cause 4 — The "I inferred organize_event but no event_id" dead-end
In handle_mention(), when the action needs an event ID but one isn't provided (very common in normal group chat), the bot returns:
❌ I inferred you want to **confirm**, but I'm not sure which event.
This is a cliff edge. Users who just described a complex negotiation ("let's do it Saturday if Ahmed and Sara can make it") get an error asking them to type confirm event 12. The whole point of the AI mode is to avoid that.
Fix: When event_id is missing for join/confirm/cancel, instead of erroring, query the database for the user's most recent active events and offer them as inline buttons. Let the user tap to apply the action — zero typing required.

async def _offer_event_selection(
    message, context, action_type: str, requester_id: int
) -> None:
    """Query active events and offer them as tap targets."""
    async with get_session(settings.db_url) as session:
        result = await session.execute(
            select(Event)
            .where(Event.state.in_(["proposed", "interested", "confirmed"]))
            .order_by(Event.event_id.desc())
            .limit(5)
        )
        events = result.scalars().all()

    if not events:
        await message.reply_text("No active events found. Use @bot to create one.")
        return

    action_label = {"join": "✅ Join", "confirm": "🎯 Confirm", "cancel": "❌ Cancel"}.get(action_type, action_type)
    buttons = [
        [InlineKeyboardButton(
            f"{action_label} #{e.event_id}: {(e.description or '')[:30]}",
            callback_data=f"mention_{action_type}_{e.event_id}"
        )]
        for e in events
    ]
    await message.reply_text(
        f"Which event did you mean?",
        reply_markup=InlineKeyboardMarkup(buttons)
    )
